## API shape

The dashboard uses the typed API routers for projects, environments, resources, deployments, servers, Swarm, Docker, web server, integrations, backups, schedules, notifications, tags, audit logs, and UpGal. Requests are organization-scoped and pass through the same capability checks as the dashboard.

For machine clients, prefer an organization API key with the smallest required permissions. Send the key as `x-api-key` or as a Bearer credential where the route accepts it. Never place a key in a repository, browser bundle, or URL query string.

## Git webhooks

Git provider webhooks are used to queue deployments for eligible Git-backed resources. Validate the provider secret, identify the organization/resource mapping, and let Upstand record the source revision. If a webhook is accepted but no deployment appears, inspect Requests, Deployments, and the provider configuration before retrying.

## SCIM

SCIM 2.0 is available at `/api/scim/v2.0/<organizationId>`. Create a provisioning token in the SCIM settings page and configure the IdP with that base URL and token. SCIM manages organization membership and profile/group state; deactivating a user revokes their organization access while preserving the global user identity.

Keep the SCIM endpoint private to the IdP integration, rotate provisioning tokens during incident response, and verify that the IdP's group-to-role mapping matches the capabilities you intend to delegate.

## MCP

UpGal exposes `/api/mcp` for JSON-RPC MCP clients. The supported protocol operations include `initialize`, `tools/list`, and `tools/call`. Authenticate with an organization API key that has MCP permissions. The server reuses organization and capability authorization; read tools can return results, while mutating operations return an approval-required result that must be completed in the dashboard.

For step-by-step editor and client configuration guides (VS Code, Claude Desktop, Claude Code, Cursor, Codex, Windsurf), see the [MCP Server Setup & Editor Integration guide](/docs/operations/mcp-server).

## Rate limits and retries

Public and authenticated API paths are rate-limited. Treat `429` responses as back-pressure: retry with exponential delay and jitter, keep requests idempotent where possible, and do not create duplicate deployments or schedules while an earlier request is still pending. Use the request/observation pages and audit log as the source of truth for asynchronous work.
