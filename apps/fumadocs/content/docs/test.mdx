## Production Installation

Upstand is designed to be installed as a Docker Stack on a Docker Swarm manager node.

### System Requirements
- **Host OS**: Ubuntu 22.04 LTS or Debian 12 (Bookworm).
- **Docker**: Engine version 24.0.0+ with Swarm mode enabled.
- **System Utilities**: `curl`, `openssl`, `git`, and `rclone` (all bundled in the official server image).

### One-Command Setup
Execute the following bootstrap command on your server manager node:

```bash
curl -fsSL https://raw.githubusercontent.com/upstand/upstand/main/install.sh | sudo bash
```

The script will:
1. Verify system dependencies and install the Docker Engine if missing.
2. Initialize Docker Swarm (if not active).
3. Generate secure random passwords as `/etc/upstand/secrets/*` files and keep `/etc/upstand/.env` limited to non-secret configuration.
4. Create the `upstand-network` overlay network.
5. Launch the Upstand Stack (`postgres`, `redis`, `server`, `web`, and `fumadocs`).

## Managing Docker Swarm Nodes

Upstand automates Swarm clustering. You can view, add, or configure worker nodes directly from the control panel.

### Joining a New Worker Node
To expand your cluster and increase build concurrency across multiple physical servers:
1. Navigate to the **Docker Swarm** tab in the dashboard.
2. Copy the Worker join token from the Manager node settings.
3. SSH into your target server and run the join command:
   ```bash
   docker swarm join --token <JOIN_TOKEN> <MANAGER_IP>:2377
   ```
4. The node will register automatically in Upstand.

## Reverse Proxy Configuration

Upstand orchestrates a central **Caddy Web Server** container to handle SSL certificates and domain proxy routing.

- **Atomic configuration**: Upstand compiles every resource mapping into one Caddyfile, formats and validates it inside the Caddy container, then performs Caddy's zero-downtime reload. If validation or reload fails, the previous known-good file is restored.
- **Persistent certificates**: Caddy's `/data` and `/config` directories are Docker-managed volumes, so ACME accounts, certificates, and renewal state survive container recreation.
- **Automatic HTTPS**: A normal hostname uses Caddy Automatic HTTPS. Caddy obtains and renews a certificate from Let's Encrypt (with Caddy's normal issuer fallback) and redirects HTTP to HTTPS. The public host must resolve to the Swarm manager and permit inbound TCP **80** and **443**; HTTP/3 additionally uses UDP 443.
- **Ports Mapping**: Caddy always publishes public 80/443, forwarding them to its configured internal listener ports. Resource targets stay private on the attachable `upstand-network` overlay (for example, `web:3000`).
- **Paths and snippets**: A resource domain can route a path prefix, strip or rewrite that prefix, and import named Caddy snippets defined in Web Server settings. These are native Caddy snippets, not Traefik middleware references. Compose resources must select the precise deployed Swarm service name.

## Backup Configurations with S3

Ensure disaster recovery by creating automated S3 destinations:
- **Encryption**: Credentials are encrypted at rest using AES-256-GCM.
- **Rclone Integration**: Bucket checks use `rclone ls` under the hood. To test connection validation manually, run:
  ```bash
  rclone ls :s3:<bucket-name> --s3-provider="AWS" --s3-access-key-id="<access-key>" --s3-secret-access-key="<secret-key>" --s3-endpoint="<endpoint>"
  ```
