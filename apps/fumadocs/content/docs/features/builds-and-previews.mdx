## Build types

Application build configuration is validated against one of six types:

| Type | Important options |
| --- | --- |
| Dockerfile | Build path, Dockerfile path, context path, target stage, build arguments, no-cache, cache cleanup |
| Railpack | Build path and pinned Railpack version |
| Nixpacks | Build path and optional publish directory |
| Heroku Buildpacks | Build path and Heroku stack version (`24` or `26`) |
| Paketo Buildpacks | Build path |
| Static | Build path, required publish directory, SPA fallback |

Paths are relative to the source checkout. Keep the build path, Docker context, Dockerfile, and publish directory inside the repository checkout.

Build secrets are distinct from ordinary build arguments. Use a secret when a value must be available during a build without becoming ordinary runtime configuration or a final image layer.

## Source triggers

Applications and Git-backed Compose resources can react to provider webhooks. Configure whether the trigger is a branch push or a tag, an optional tag pattern, and watched paths. A webhook records the source commit and queues a deployment for the resource. The deployment history is the authoritative record of the commit that was built.

Use the provider-specific webhook secret configured in the Git provider integration. Do not put provider tokens in repository files.

## Queues and concurrency

Deployments are queued per server and serialized per resource. The worker acquires a resource lock before building so two builds cannot update the same workload concurrently. Build concurrency can be adjusted for a server; databases cannot be used as application build-concurrency hosts.

The Observation deployment view shows queued, running, successful, and failed work. Inspect the deployment's stage output before retrying or changing the resource.

## Preview deployments

Applications can enable preview deployments with a limit, wildcard, HTTPS option, and preview port. Preview settings are resource configuration; they do not replace a production environment. Keep the preview limit bounded and ensure the wildcard DNS record points at the routing server when using preview hostnames.

## Rollback

An application can be redeployed from a historical source revision. When rollback registry settings are configured, the service update can use the designated encrypted registry credentials for the rollback image path. Review the deployment history and current environment variables before rolling back: a code rollback does not automatically undo unrelated configuration changes.
